# Alert Bot

Телеграм-бот для двух задач:

1. **Воздушные тревоги** — мониторит канал-источник и при сообщениях о начале тревоги / отбое публикует заранее заданный шаблон.
2. **Отключения электроэнергии** — мониторит канал энергетиков (Севастопольэнерго), с помощью LLM извлекает график веерных отключений (очередь + время) и публикует красиво оформленный пост с фиксированным списком адресов.

Обе задачи публикуют **параллельно в Telegram и в MAX Messenger**. Юзербот только **читает** каналы, публикация идёт через официальный **Bot API** (гибридный режим — безопаснее для аккаунта).

---

## Возможности

- Мониторинг двух источников с независимыми циклами (энергоканал не влияет на скорость выдачи тревог).
- **Тревоги/отбой:** сопоставление по ключевым фразам (фраза в начале текста, игнорируются эмодзи, регистр, `!`) → публикация статичного шаблона.
- **Энергоканал:** LLM-извлечение структуры (тип / очередь / время) + сверка с текстом поста (защита от ошибок модели) + подстановка фиксированных адресов очереди.
- Параллельная публикация в Telegram + MAX; если один канал упал — второй всё равно получит.
- Защита от дубликатов (обработанные ID хранятся по каждому каналу отдельно).
- Автопереподключение к Telegram, пропуск устаревших постов после рестарта, уведомления администратору о сбоях.

## Как это работает

### 1. Воздушные тревоги
1. Каждые ~30 сек бот читает последние посты канала-источника (`SOURCE_CHANNEL`).
2. Текст нормализуется (убираются эмодзи, `!`, регистр, лишние пробелы).
3. Если текст **начинается** с фразы из `keywords_alert.txt` → публикуется шаблон `ALERT_TEMPLATE`; из `keywords_clear.txt` → `CLEAR_TEMPLATE`.
4. Шаблон уходит в `TARGET_CHANNEL` (Telegram) и в MAX.

### 2. Отключения электроэнергии
1. Отдельный цикл читает канал `ENERGY_SOURCE_CHANNEL`.
2. Текст нового поста отправляется в LLM (через OpenAI-совместимый шлюз, напр. vsellm.ru), который возвращает строгий JSON:
   `{"type": "outage|lifted|ignore", "confirmed": bool, "windows": [{"queue": 1|2, "from": "ЧЧ:ММ|null", "to": "ЧЧ:ММ"}]}`
3. Код **валидирует** и **сверяет** извлечённые очередь/время с текстом поста. Не сошлось — пост не публикуется, а черновик уходит администратору в личку.
4. Публикуется **только `outage`** (веерное отключение по графику очереди):
   - `confirmed=true` → «🔌 Отключение электроэнергии — N очередь»
   - `confirmed=false` → «⚠️ Ожидается отключение электроэнергии — N очередь»
   - к посту добавляется фиксированный список адресов этой очереди (`data/queue_N_addresses.txt`) и подпись источника.
5. Игнорируются (не публикуются): снятие/восстановление, технологические нарушения и аварии, плановые/ремонтные работы по адресам, локальные отключения без очереди, новости и реклама.

> Итоговый пост собирает **код**, а не LLM — модель отвечает только за извлечение полей. Это исключает попадание «галлюцинаций» в готовый текст.

## Архитектура

```
main.py — AlertBot: два независимых asyncio-цикла
  ├─ цикл тревог      → process_new_posts   (SOURCE_CHANNEL)
  └─ цикл энергоканала → process_energy_posts (ENERGY_SOURCE_CHANNEL)   [фоновая задача]
  └─ src/
     ├─ config.py           — конфигурация из .env + validate()
     ├─ parser.py           — чтение постов (telethon) с таймаутами
     ├─ filter.py           — TextFilter: нормализация + тревога/отбой
     ├─ llm_extractor.py    — LLMExtractor + verify_outage (энергопосты)
     ├─ energy_formatter.py — сборка поста об отключении + адреса
     ├─ bot_publisher.py    — публикация в Telegram через Bot API
     ├─ publisher.py        — публикация через userbot (запасной путь)
     ├─ max_publisher.py    — публикация в MAX Messenger
     ├─ storage.py          — обработанные ID по каналам (+ asyncio.Lock)
     └─ logger.py           — логирование
```

**Изоляция:** энергоканал крутится в отдельной фоновой задаче. Сетевой запрос к LLM (`await`) уступает управление, поэтому медленный/зависший вызов модели **не задерживает** проверку тревог. Общий файл обработанных ID защищён локом.

## Установка

```bash
cd alert_bot
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # затем заполнить .env
```

При первом запуске потребуется авторизация userbot: номер телефона → код из Telegram → при необходимости пароль 2FA.

## Конфигурация (`.env`)

### Telegram / userbot
| Переменная | Описание |
|---|---|
| `API_ID`, `API_HASH` | ключи с https://my.telegram.org/apps |
| `PHONE` | номер телефона userbot (с кодом страны) |
| `SOURCE_CHANNEL` | канал-источник тревог (напр. `@razvozhaev`) |
| `TARGET_CHANNEL` | целевой канал публикации (напр. `@nts_novosti`) |
| `CHECK_INTERVAL` | интервал проверки, сек (по умолчанию 30) |
| `RANDOM_DELAY`, `DELAY_VARIANCE` | случайный разброс интервала |

### Публикация
| Переменная | Описание |
|---|---|
| `BOT_TOKEN` | токен бота от @BotFather (гибридный режим — публикация через Bot API) |
| `ADMIN_CHAT_ID` | Telegram-ID администратора для уведомлений о сбоях и черновиков |
| `MAX_BOT_TOKEN` | токен бота MAX (пусто — MAX отключён) |
| `MAX_TARGET_CHANNEL` | chat_id целевого канала MAX |

### Шаблоны тревог
| Переменная | Описание |
|---|---|
| `KEYWORDS_ALERT_FILE`, `KEYWORDS_CLEAR_FILE` | файлы ключевых фраз |
| `ALERT_TEMPLATE`, `CLEAR_TEMPLATE` | тексты шаблонов (HTML: `<b>`, `<i>`) |

### Энергоканал (второй источник)
| Переменная | Описание |
|---|---|
| `ENERGY_SOURCE_CHANNEL` | канал энергетиков (напр. `@sev_energo`); пусто — функция отключена |
| `LLM_API_KEY` | ключ LLM-шлюза (напр. vsellm.ru) |
| `LLM_BASE_URL` | базовый URL, по умолчанию `https://api.vsellm.ru/v1` |
| `LLM_MODEL` | модель, напр. `openai/gpt-4.1-mini` |
| `QUEUE_1_ADDRESSES_FILE`, `QUEUE_2_ADDRESSES_FILE` | файлы адресов по очередям |
| `ENERGY_SOURCE_NAME` | подпись источника в постах (текстом) |
| `ENERGY_LIFTED_ENABLED` | публиковать посты о снятии ограничений (по умолчанию `false`) |

> Энергоканал включается только если заданы **и** `ENERGY_SOURCE_CHANNEL`, **и** `LLM_API_KEY`.
> Рекомендуемая модель — не «думающая» (напр. `gpt-4.1-mini`): reasoning-модели тратят лимит токенов на внутренние рассуждения и могут вернуть пустой ответ.

## Запуск

**Как сервис (systemd):**
```bash
sudo systemctl restart alert_bot
sudo systemctl status alert_bot
```

**Вручную:**
```bash
source venv/bin/activate
python3 main.py
```

## Логи

Пишутся в `logs/bot_YYYYMMDD.log` (имя — по дате запуска процесса) и в stderr сервиса.

Полезные выборки:
```bash
# события энергоканала
grep -i энергопост logs/bot_$(date +%Y%m%d).log
# ошибки/сбои
grep -iE "ошибк|traceback|невалидн" logs/bot_$(date +%Y%m%d).log
```

## Обновление на сервере

```bash
cd ~/projects/bots/alert_bot
git pull
# при изменении зависимостей:
./venv/bin/pip install -r requirements.txt
sudo systemctl restart alert_bot
```

Файлы `.env`, `*.session`, `data/processed_posts.json` в `.gitignore` — при обновлении не затрагиваются. Формат `processed_posts.json` мигрирует автоматически.

## Безопасность

- Никогда не коммитьте `.env`, session-файлы и токены (уже в `.gitignore`).
- `API_HASH` и session-файлы содержат данные авторизации — храните в секрете.
- Списки адресов и шаблоны (`data/*.txt`) можно править без изменения кода — после правки перезапустите сервис.
